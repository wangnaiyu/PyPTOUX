面向 A5 的算子开发与调优体验设计

A5 / Ascend 950 在 Cube-Vector 算力配比、Mix 子图、显式同步、低精度数据表达等维度发生了 代际级变化。本报告从 A5 架构特点出发，梳理算子开发全流程在新硬件下的用户痛点与产品机会， 并给出以"调优工程师主场景"为切入的设计方案与三个高光设计点。

01 · 引子

一份慢 2.3× 的 trace

调优工程师拿到一份 A5 上的运行 trace，会话 glm_attention.py · #c4a7e21。 编译成功，但 wall-clock 比预期慢 2.3 倍。

他要回答的不是「哪里慢」，而是「为什么慢」—— 是 tile 失配？dtype 选错？ 还是 Mix 子图边界拆裂带来的 Sync Wait？

回答这个问题，他过去需要打开 swimlane · Block Graph viewer · 编译日志 · 代码 IDE —— 在四个工具间手工 grep，靠经验把碎片拼回去。这份报告，就是从这一刻开始的。

05 · 设计原则

新一代工具 ≠ 老工具加新功能

在进入方案之前，明确三条立场。这三条不仅指导 A5 上的设计选择，也是和顶层设计文档 （Simpler runtime redesign）同向的产品立场。

01

从「看现象」到「指根因」

过去的工具大多是仪表盘，把数据画出来就交差。新一代工具必须给出"为什么"的可解释路径，而不是只告诉你"哪里慢"。

仪表盘 → 诊断闭环
02

从「按图分割」到「按定位链贯通」

swimlane / Block Graph / 源代码 / 硬件路径 不应是四个孤岛。一次点击应能穿透完整定位链，可信度分层始终在场。

四工具 grep → 一键穿透
03

从「专家私货」到「团队资产」

调优经验不能只留在个人 trace 和 IM 里。Contribute Back 让一次诊断成为团队 Playbook，让下一次同类问题被工具主动识别。

单兵英雄 → 集体智慧

07 · 高光设计点

三个不可替代的差异点

不罗列十几个细节。从全部设计中挑出三个最有冲击力的：每一个都解决一个无法靠"老工具加功能" 解决的根本问题。

🅰 · HIGHLIGHT 01

跨视图定位链一键穿透

swimlane → rootHash → callOpMagic → leafHash → wrapId → Graph / AIC / AIV / GM / L2。 过去要打开 4 个工具、做 4 次 grep；现在是详情面板里的 6 个跳转按钮。

它解决了：A5 上"定位链长 + 跨视图工具割裂"这一最高频的痛点；调优工程师 / 算子开发者 / 新手开发 三个角色共同受益。
🅱 · HIGHLIGHT 02

可信度分层 UI

verified / inferred / article-claim 三档线型与卡片样式，贯穿硬件图 · PMU 详情 · 报错提示 · Playbook 沉淀。每一条事实都自带"我有多确定"的视觉标识。

它解决了：BufferID / URMA / CCU 等公开材料宣称与 trace 实证混淆的根本风险；让工具"诚实"成为可被信任的产品基石。
🅲 · HIGHLIGHT 03

Contribute Back 沉淀回路

一次个人诊断 → 团队 Playbook → 下一次进入 swimlane 时的自动推荐。配合顶层设计的 Replay 模式，调优会话本身可以被原样重播。

它解决了：调优经验依赖单兵英雄的根本问题，让 Tech Lead 与"团队整体"成为新一代工具的真正用户。
