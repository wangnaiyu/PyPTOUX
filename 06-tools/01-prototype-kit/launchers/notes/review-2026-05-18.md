## 2. A · 结构与归类

- 放在 `06-tools/01-prototype-kit/launchers/` 合适。它解决的是“本地预览启动/停止”，不是外发打包，也不是具体 prototype 源码。
- 当前两个 launcher 暂时可以接受重复 shell 模板；新增第三个及以后，建议抽出公共生成逻辑，而不是手工复制。优先方向不是让 `.command` source 一个 shared shell 文件，而是用一个 registry 生成 `.command` 或让通用 launcher 读取 registry 后启动指定条目。
- 命名规则 `preview-<topic>-<prototype>.command` 符合内部归档习惯。当前 `preview-pypto-studio-v1.command`、`preview-a5-pmu-group2-loop.command` 都清楚。A5 文件名没有带目录编号 `06-` 是可接受的，因为 launcher 名面向人读；路径和端口表已经保留目录编号。
- `launchers/notes/spec.md` 已经承担了 toolkit 契约说明职责。README 也足够让第一次使用者上手。

## 3. B · 事实正确性

- `cd /tmp` + `python3 -m http.server --directory "$ROOT"` 的组合合理，可以避开 Finder/Terminal 启动时 cwd 不稳定，以及中文/空格路径导致的 cwd 问题。
- 双路径模型的动机合理：`.claude/launch.json` 已明确说 helper 沙箱无法读取 `~/Documents` 下的 `.command`，因此用户双击路径和 Claude Code preview 路径暂时分开。这里属于本仓库已验证约束，不建议为了“优雅”强行合并。
- spec §6 的 `/tmp/pyptoux-preview/<name>/` 描述与 `.claude/launch.json` 有一个小不一致：`name=a5-pmu-group2-loop`，但目录是 `/tmp/pyptoux-preview/06-a5-pmu-group2-loop`。建议要么改 spec 为“镜像目录由 launch 配置指定，不保证等于 name”，要么统一目录名。
- spec §7 说首次运行 macOS 可能弹防火墙“允许接受网络连接”。因为 server 绑定 `127.0.0.1`，外部不可访问；是否弹窗取决于系统策略和 Python 签名状态。建议保留为低概率提示，不要写成必然。
- spec §4 说 stop 只 kill 注册端口、“不会误伤其他本地 http server”不完全准确。它不会扫描全部 server，但如果其他本地服务刚好占用 8766/8770，仍会被 kill。建议后续用 `lsof` 结果校验命令行包含 `http.server` 和预期目录，或者在 PID 文件里登记 launcher 启动的 PID。

## 4. C · 接口约定与端口

- 8766 / 8770 作为本地 prototype 预览端口可接受。它们不属于常见前端默认端口如 3000、5173、8000、8080、8888，也避开了常见的 8765 示例端口。建议把 8770-8799 作为 PyPTOUX prototype preview 的本地保留段；8766 已经使用，可以继续保留但以后从 8771 递增。
- 每个原型固定端口比动态端口更适合 demo、README、截图和 Claude preview registry。动态端口会让“同一个原型固定 URL”消失，不利于协作和复现。
- 端口注册表建议升级为可解析配置。优先 JSON，不建议 YAML：macOS/bash 环境下没有稳定 YAML parser，而 Python 标准库可直接读 JSON。可以考虑 `launchers/registry.json`：

```json
{
  "pypto-studio-v1": {
    "port": 8766,
    "root": "05-prototypes/05-pypto-studio/experiments/v1"
  },
  "a5-pmu-group2-loop": {
    "port": 8770,
    "root": "05-prototypes/06-a5-pmu-visualization/experiments/group2-loop"
  }
}
```

- 有了 registry 后，`stop-previews.command` 可以从 registry 读取端口，README/spec 只展示表格生成结果或人工摘要，减少三处同步风险。
- `list-previews.command` 值得做，但优先级低于 registry。它可以在 registry 之后自然实现：列出 port、root、是否 LISTEN、PID。
- 不建议 pin `python3` 到固定绝对路径。当前 `/usr/bin/env python3` 更适合用户机器差异；更需要的是启动前 preflight：找不到 python3 时给出清晰错误，提示安装 Xcode Command Line Tools 或 Homebrew Python。

## 5. D · share-safe 与 bundler 的边界

- launcher 不应承担 share-safe 判断。它只服务本地预览，README 里“外发请用 prototype-bundler”这句正确。
- 需要再补一句：即使用 bundler 打包，L3 / exploration-only 原型仍不能因为“可双击打开”就外发；外发需要遵守 `AGENTS.md` / `CLAUDE.md §5` 的 Codex + Claude 双通过。
- 如果未来 launcher 集成 bundler 的二选一入口，“导出单文件 HTML”按钮应跳到 bundler 的明确流程，不要在 launcher 内重复实现 share-safe UI。

## 6. E · 其它边界与风险

- 建议新增 PID 文件或签名校验，避免 `stop-previews.command` 误杀同端口的非 PyPTOUX 进程。最轻量做法是在 `/tmp/pyptoux-preview-pids/<name>.pid` 写入 PID，stop 时先读 PID 并校验端口。
- 当前两个 launcher 都会 `open` 浏览器后再启动 server，中间用 `sleep 1`。如果机器慢，浏览器可能先打开空连接再自动刷新失败。问题不大，但更稳的方式是先启动 server，再后台轮询端口 ready 后 open；这需要不再用简单 `exec`，可后续再做。
- `.claude/launch.json` 与 launcher registry 长期会 drift。若引入 `registry.json`，建议 `.claude/launch.json` 仍可保持内联 shell，但由 Codex/Claude 更新时以 registry 为 source of truth。
- README 对第一次使用者足够清楚；建议在“新增一个 launcher”的步骤里补“同步 `.claude/launch.json` 仅在需要 Claude preview 时才做”，避免用户以为每次都必须动 Claude 配置。
