# Codex Review · 2026-05-18

## 0. 范围与结论

本次 review 只读了 `notes/spec.md`、`bundler.py`、`prototype-bundler.command` 和 `README.md`，没有直接修改实现或 spec。

总体判断：`prototype-bundler` 放在 `06-tools/01-prototype-kit/converters/` 合适，`notes/spec.md` 已经覆盖了新增 toolkit 应有的用途、输入输出、命令入口、边界和待 review 项。当前实现已经能成功打包 A5 PMU Group 2 的 ES module 原型；但 spec 与实现之间有几处需要收敛，尤其是 share-safe 语义、远程/不支持资源的校验、top-level await / `import.meta` 的描述，以及 `--infer-name` 的接口稳定性。

我没有发现必须立刻由 Codex 越界修复的实现错误；以下按建议交给 Claude 后续吸收。

## 1. 已做检查

- `bash -n prototype-bundler.command` 通过。
- `python3 -B bundler.py --infer-name 05-prototypes/06-a5-pmu-visualization/experiments/group2-loop` 输出：`06-a5-pmu-visualization-group2-loop` + `05-prototypes/06-a5-pmu-visualization/experiments`。
- `python3 -B bundler.py --cli 05-prototypes/06-a5-pmu-visualization/experiments/group2-loop -o /private/tmp/pyptoux-a5-pmu-bundle-test.html` 成功。
- `python3 -B bundler.py --cli 05-prototypes/05-pypto-studio/experiments/v1 -o /private/tmp/pyptoux-studio-bundle-test.html` 失败，原因是 PyPTO Studio v1 使用 classic `<script src=...>`，没有 `<script type="module" src=...>`。这符合当前 spec 的 ES module 适用范围，但 README 可以更明确地说它不能打包 PyPTO Studio v1 这种 classic-script 原型。
- 本机环境：`/usr/bin/python3` = Python 3.9.6，`tkinter.TkVersion` = 8.5，`/bin/bash` = 3.2.57。

## 2. A · 结构与归类

- 放在 `converters/prototype-bundler/` 合适。它的主要职责是把一种原型形态转换成另一种交付形态，不是 launcher，也不是 shared runtime。
- 暂时不必拆到 `shared/`。目前只有一个 converter 使用 HTML/CSS/JS 解析逻辑，过早抽公共层会增加维护成本。等第二个 converter 或 launcher 也需要复用路径推断、share-safe banner、build metadata 时，再抽 `shared/prototype-artifact.py` 或类似工具更稳。
- 命名规则 `<topic>-<prototype>` 符合内部归档习惯，尤其适合 `05-prototypes/<numbered-topic>/experiments/<prototype>/`。不建议默认带日期；日期适合作为 build log 里的 metadata，而不是文件名的一部分。若同一 prototype 多次外发，可以用用户手改文件名或后续 `--variant` / `--version` 参数解决。
- `--infer-name` 输出 tab 分隔对当前 `.command` 够用，但不是长期稳定接口。路径理论上可以包含 tab，未来也可能需要返回 `topic`、`prototype`、`defaultOutputDir`、`warnings`。建议新增一行 JSON 输出模式，例如 `--infer-name-json`，保留 tab 输出作为 legacy，避免破坏现有 `.command`。

## 3. B · 事实正确性

- `file://` 下 ES module 跨文件加载会被浏览器同源策略拦住，这个问题判断准确。
- macOS / Tk 的方向判断基本准确：本机 Apple Python 3.9.6 确实绑定 Tk 8.5；Python.org 的 macOS Tcl/Tk 文档也建议避免 Apple 8.5.9 一类老 Tk 问题，并说明官方 Python.org 安装器会带私有 Tcl/Tk 8.6。建议 spec 把“macOS 自带 Python 3.9 绑的是 Tcl/Tk 8.5”改成“本机 / Apple-provided Python 3.9.6 环境下是 Tk 8.5；Python.org installer 通常是 8.6”，避免说成所有 macOS 的绝对事实。参考：https://www.python.org/download/mac/tcltk/
- bash 3.2.57 的兼容性判断准确，本机 `/bin/bash` 即 3.2.57。`.command` 用 `#!/usr/bin/env bash` 时，如果用户 PATH 前面有 Homebrew bash，会跑新版 bash；这通常不是问题，但 spec 里“Finder 双击启动时进的是系统自带 `/bin/bash 3.2.57`”建议改成“按当前 shebang 与 PATH，至少需要兼容系统 bash 3.2”。本机事实仍成立。
- spec §10 说“macOS only（依赖 macOS 系统 Python3 + tkinter）”与当前设计冲突。GUI 已改成 osascript，不依赖 Python GUI 库；这里应改成“不依赖 tkinter，仅 CLI 依赖 Python3，GUI 依赖 osascript”。
- share-safe 的语义不能简化为“单文件 + 无 Google Fonts”。按 `AGENTS.md` / `CLAUDE.md §5`，share-safe 同时需要运行时无远程依赖、数据真实性/边界/结构通过 Codex、前端呈现质量通过 Claude。当前 spec 说 HTML 注释只是占位，这个方向是对的。

## 4. C · 接口约定

- 建议把 CLI 内部 API 分成两层：`build_bundle(...)` 保持纯函数式产物生成；`run_cli(...)` 负责 stderr/stdout 和 exit code。目前已经基本做到，结构健康。
- `--infer-name` 的 tab 输出可继续保留，但建议加 JSON 模式。`.command` 用 shell 切 tab 是可行的，不过 JSON 更适合后续加字段，也减少中文/空格/特殊字符路径的解析风险。
- 需要明确 classic scripts 是否永远不支持。现在 `05-prototypes/05-pypto-studio/experiments/v1/` 不能打包，这是合理边界，但 README 的“纯前端原型”可能让使用者误以为 classic `<script src>` 也可用。建议 README 在“不适用”里补一句“classic 多脚本原型暂不支持，例如 PyPTO Studio v1”。
- 当前 import/export 改写有两个值得补测试的边界：
  - `import { foo as bar } from './x.js'` 现在会生成 `const { bar } = ...`，但正确语义应是 `const { foo: bar } = ...`。
  - `export { foo as bar }` 现在可能生成 `__exports.bar = bar`，但本地变量名是 `foo`。
  现有 A5 原型可能没踩到，但这是 parser 级边界，应写入 spec 限制或补实现。

## 5. D · share-safe 流程

- 不建议用“强制勾选已确认可外发”作为保存门槛。工具无法判断 L1/L2/L3、截图/文本是否可外发、前端质量是否通过；一个 checkbox 容易制造“已经审计”的错觉。
- 更推荐两步：
  1. 默认产物继续写 `level : L3 demo · exploration-only`，并在 GUI 成功页提示“这不是 share-safe 审批”。
  2. 后续新增显式参数或对话框选项：`exploration-only` / `share-candidate`。只有用户选择 `share-candidate` 时，才要求确认“已完成 Codex 数据/结构 review + Claude 前端质量 review”，并把确认写进 metadata，而不是悄悄去掉 L3 标记。
- build metadata 建议保留 HTML 头注释，同时可选写 `05-prototypes/<topic>/notes/build-log.md`。默认每次打包都写 notes 可能让工具产生很多日志噪音；建议做成 `--write-build-log` 或 GUI checkbox。进入外发候选时再写 build log 更有价值。

## 6. E · 其它边界与风险

- spec §6 说遇到绝对路径、远程 src、Web Worker 等会失败，但实现现在对远程 CSS / module script 是保留原 tag，不报错。这个差异会影响 share-safe 判断：保留远程资源意味着产物不是离线稳定可用。建议要么实现校验失败，要么 spec 改成“保留但标记 warning，不视为 share-safe”。
- `top-level await` 和 `import.meta` 不能说被 IIFE 保留。当前实现把每个模块体放进普通 function IIFE，top-level await 在 function 内会变成语法错误，`import.meta.url` 的含义也不等同原模块文件。spec §6 这句需要修正为“不支持 / 未保证”。
- side-effect import（`import './setup.js'`）当前不会被 `_IMPORT_ANY_RE` 收集，也不会被 `_strip_imports_and_export_keywords` 删除，打包后仍可能留下 import 语句并触发 `file://` 问题。建议明确不支持并报错，或补解析。
- CSS 里的 `url(...)` 资源不会内联；如果字体、图片在 CSS 中被引用，单文件 HTML 仍可能缺资源。建议 spec 加进首版限制。
- 相对路径 `../` import / CSS href 当前可解析到 prototype 根目录外，存在把相邻文件意外打进外发 HTML 的风险。建议 share-safe 模式限制所有被内联资源必须位于所选 prototype folder 内。
- README 对第一次使用者足够清楚，但“我打包后能给谁？”这一节建议更强硬：L3 产物默认只能内部流转，不能外发给业务方/产品方/设计师/用户，除非另有 Codex + Claude review 记录。
