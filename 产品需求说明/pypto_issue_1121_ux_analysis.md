# PyPTO Issue #1121 — UX 深度分析：Mix 子图泳道图核间同步可视化

> 基于 [hw-native-sys/pypto](https://github.com/hw-native-sys/pypto) 仓库信息与 Issue 设计方案，从用户体验（UX）视角进行深入分析。

---

## 1 用户画像与使用场景

### 1.1 目标用户

根据 PyPTO 仓库 README 的分层抽象设计，泳道图功能的用户可细分为三类：

| 用户角色 | 核心诉求 | 使用泳道图的频率 | 关注粒度 |
|---------|---------|----------------|---------|
| 算法开发者 | 快速验证算法正确性，粗粒度了解性能 | 低 — 仅在发现性能异常时查看 | Tensor 级整体耗时 |
| 性能优化专家 | 定位瓶颈、优化核间负载均衡 | 高 — 日常工具 | Tile/Block 级逐核耗时 |
| 系统开发者 | 调试调度策略、验证 MPMD 执行行为 | 中 — 集成联调阶段 | Block 级核间通信与同步 |

Issue #1121 的核心受益者是**性能优化专家**，但 UX 设计必须兼顾其他两类用户的认知负担。

### 1.2 典型场景

PyPTO 采用 MPMD 执行模式，多个核心并行执行不同的程序片段。在 mix 子图场景下，一组 task 跨多个核执行并存在同步依赖。当前泳道图只展示了每个 leaffunction 的起止时间，无法区分"真正在计算"和"在等待同步"。

用户的典型思考路径如下：

```
"Core 3 的这段 task 为什么耗时 200μs？" 
  → 查看泳道图 → 只看到一个完整的 bar 
  → 无法判断：是计算慢？还是在等 Core 1 的结果？
  → 无法下手优化
```

期望达到的体验是：

```
"Core 3 的 task 耗时 200μs" 
  → 打开 mix 模式 → 看到 bar 被分为「计算段」和「同步等待段」
  → 同步等待段有连线指向 Core 1 的对应 SyncSet 点
  → 一眼看出：计算只用了 50μs，剩下 150μs 都在等同步
  → 优化方向清晰：减少 Core 1 的前序计算量
```

## 2 现有泳道图的 UX 问题诊断

### 2.1 信息缺失

当前泳道图由 `merged_swimlane.json` 驱动，数据来源于 `tilefwk_L1_prof_data`。PyPTO 的工具链文档提到，完整的编译产物和运行时性能数据可通过 IDE 集成工具链可视化以识别性能瓶颈。但在 mix 子图场景下，存在以下信息缺口：

- **同步等待不可见**：task bar 呈现为单一连续色块，用户无法区分计算时间与等待时间
- **核间依赖不可见**：无法看到"Core A 的 SyncWait 在等 Core B 的 SyncSet"
- **分组关系不可见**：属于同一组 mix 子图的 task 散落在各核泳道中，没有视觉聚合

### 2.2 认知负担过高

- 用户需要在多个核的泳道之间**来回扫视**，心算哪些 task 属于同一组 wrap
- 需要手动对齐时间轴来推断同步关系
- 缺乏"入口"——不知道从哪里开始分析一组 mix 任务

### 2.3 模式切换成本

Issue 中提到新增 "mix 显示模式"，这意味着用户需要主动发现并切换模式。如果入口不明显或切换后界面变化剧烈，会产生方向感丧失（disorientation）。

## 3 UX 设计建议

### 3.1 渐进式披露（Progressive Disclosure）

不应让所有核间同步信息一次性涌现。建议采用三层渐进结构：

**第一层：默认视图 — 染色标记 + 同步占比指示器**

在不打开 mix 模式的情况下，对现有泳道做最小改造：

- 对属于 mix 子图的 task bar，在右侧添加一个微型占比指示器（类似 mini sparkline），显示"同步等待占比"
- 用暖色（如琥珀色）标记同步等待比例超过 30% 的 task，作为视觉锚点吸引注意

这层改造不需要打开 mix 模式，对算法开发者零干扰，对性能专家是"嗅探器"。

**第二层：Mix 模式 — 分组框 + 核间连线**

用户打开 mix 模式后：

- 同一组 wrap 的 task 用统一底色 + 虚线边框框起来（Issue 方案中的"框图"）
- 核间同步关系用带箭头的弧线连接（SyncSet → SyncWait），颜色与 wrap 底色一致
- 1C2V 格式排布使同组核在空间上靠近，缩短视觉扫描距离

**第三层：核内流水详情 — 双击展开**

双击某个 task 后展开核内流水视图（Issue 方案已有此设计），建议补充：

- 展开时用动画过渡，保持上下文方向感
- 在展开的流水视图中，用垂直虚线标记 SyncSet/SyncWait 的时间点
- 提供"返回上层"的面包屑导航

### 3.2 1C2V 排布的视觉设计

Issue 方案中提到核以 1C2V 格式排布。这是 Ascend 硬件的物理拓扑（1 个 Cube 核 + 2 个 Vector 核），映射到泳道图时需要注意：

**推荐布局**：

```
┌─────────────── Mix Group A ───────────────┐
│ Core 0 (Cube)   ████████░░░████████       │
│ Core 0 (Vec-0)  ██████░░░░░░██████████    │
│ Core 0 (Vec-1)  ░░░░░████████░░░░████     │
│                  ←sync→    ←sync→         │
│ Core 1 (Cube)   ░░░████████████░░░        │
│ Core 1 (Vec-0)  ████░░░░░████████████     │
│ Core 1 (Vec-1)  ██████████░░░░░░████      │
└───────────────────────────────────────────┘
```

设计要点：

- **分组边框**：用圆角虚线矩形包裹同组 task，边框颜色与组标签色一致
- **核心间距**：同一 1C2V 单元内的三条泳道紧凑排列（间距 4px），不同单元之间加大间距（16px），形成自然视觉分组
- **同步连线**：水平方向画弧线连接 SyncSet 和 SyncWait 时间点，使用半透明线条避免遮挡 task bar
- **配色**：计算段用组色的饱和色（opacity 1.0），等待段用同色的淡色（opacity 0.25），形成"实-虚"对比

### 3.3 同步关系的视觉编码

核间同步可视化是本 Issue 的核心 UX 挑战。推荐方案如下：

**时间维度（水平轴）**：

- 在 task bar 内部，用颜色深浅区分「计算段」和「同步等待段」
- 等待段起始处放置一个小三角标记（▼ = SyncWait 开始等），匹配处放置反向标记（▲ = SyncSet 发出信号）

**关系维度（垂直方向）**：

- SyncSet 到 SyncWait 之间画贝塞尔曲线，方向从信号发送方指向接收方
- 多条同步线存在时，用不同线型（实线/虚线/点线）或颜色区分组别
- hover 某条同步线时高亮对应的两个端点 task，同时 dim 其他元素

**交互维度**：

- 点击 task bar 的同步等待段 → 弹出 tooltip 显示"等待 Core X 的 SyncSet #N，等待耗时 XX μs"
- 在 mix 模式下点击同步连线 → 高亮这对同步关系涉及的所有 task
- 提供"只看同步路径"的过滤按钮，隐藏非 mix 的 task，聚焦核间协作

### 3.4 数据驱动的 UX 考量

Issue 方案涉及多处数据格式变更，从 UX 角度提出以下兼容性建议：

**merged_swimlane.json 扩展**：

```json
{
  "tasks": [
    {
      "core_id": 0,
      "task_id": "task_42",
      "start_us": 100.5,
      "end_us": 300.2,
      "mix_group": "group_A",
      "sync_events": [
        {
          "type": "sync_wait",
          "timestamp_us": 150.3,
          "peer_core": 1,
          "peer_event": "sync_set",
          "peer_timestamp_us": 148.0,
          "wait_duration_us": 2.3
        }
      ]
    }
  ],
  "mix_groups": [
    {
      "group_id": "group_A",
      "cores": [0, 1],
      "color_hint": "#4A90D9"
    }
  ]
}
```

设计要点：

- `sync_events` 嵌套在 task 内而非独立数组，方便前端按 task 渲染
- `mix_groups` 提供分组元数据和推荐颜色，IDE 可覆盖
- `wait_duration_us` 预计算好，前端不需要再做减法
- 向后兼容：如果 `sync_events` 字段缺失，前端回退到经典模式显示

**性能预算**：

Issue 中最大同步数暂定 64。对于一个有 N 个核、每核 M 个 task 的场景，同步连线最多 N × M × 64 条。当 N=32, M=16 时约有 32,768 条同步线。前端需要：

- 默认只渲染当前视窗内可见的同步线（虚拟化渲染）
- 提供"同步线密度"滑块，在全量展示和按阈值过滤之间平衡
- 缩放到高层级时将同步线聚合为"同步带"（类似热力图），避免视觉混乱

### 3.5 Mix 模式的入口与切换

**入口设计**：

- 在泳道图工具栏添加 "Mix" 切换按钮（类似 Chrome DevTools 的 Layout 模式切换）
- 当检测到 `merged_swimlane.json` 包含 mix_groups 数据时，按钮自动高亮提示
- 首次使用时显示一句引导文案："检测到 mix 子图同步数据，开启 Mix 模式查看核间协作关系"

**切换动画**：

- 打开 mix 模式：泳道从默认排布平滑过渡到 1C2V 分组排布，分组框淡入，同步线逐条画出（stagger animation）
- 关闭 mix 模式：反向动画，分组框淡出，恢复默认排布
- 过渡时间控制在 300-400ms，使用 ease-in-out 曲线

### 3.6 可访问性（Accessibility）

- 同步连线不仅依赖颜色，还需用线型（实线/虚线）区分不同组
- 等待段除了用低 opacity，还需加斜线填充纹理（diagonal hatch），确保色觉障碍用户可辨识
- 所有同步信息可通过键盘导航（Tab 切换 task，Enter 展开详情，Esc 返回）
- 提供纯文本模式（表格形式）作为泳道图的替代视图，方便屏幕阅读器用户

## 4 与 PyPTO 架构的对齐分析

### 4.1 编译产物链路

PyPTO 的多级计算图变换路径为：Tensor Graph → Tile Graph → Block Graph → Execution Graph → PTO 虚拟指令。Issue 中的修改点分布在这条链路的后半段：

```
Tensor Graph                    （不涉及）
    ↓ Compilation Passes
Tile Graph                      （不涉及）
    ↓ Compilation Passes
Block Graph                     （不涉及）
    ↓ PipeSync Pass
Execution Graph + Sync Info     ← Pass 模块：记录核间同步关系
    ↓ CodeGen
PTO Assembly + Timestamps       ← CodeGen 模块：插入时间戳指令
    ↓ Machine Runtime
Profiling Data                  ← Machine 模块：采集并导出
    ↓ IDE Toolchain
Swimlane Visualization          ← IDE 模块：渲染泳道图
```

**UX 启示**：同步信息从 Pass 阶段就确定了静态依赖关系，但运行时才产生实际的等待耗时。IDE 应同时展示"静态同步拓扑"（哪些核之间有同步）和"动态同步代价"（实际等了多久），让用户先理解结构再分析性能。

### 4.2 与 MPMD 调度模型的 UX 映射

PyPTO 使用 MPMD 方式将不同程序片段调度到不同核。在 UX 上，每个"程序"对应一条泳道，每个核独立执行。Mix 子图打破了这个独立性——多个核的 task 存在耦合。

泳道图的 mix 模式实际上是在**可视化 MPMD 执行中的"局部 SPMD"行为**：同一组 mix 内的核在执行协作任务，需要同步点对齐。这个概念对用户来说需要有直观的视觉隐喻：

- 独立 task = 自由泳道
- mix group = 被"栅栏"（barrier）对齐的协作泳道

建议在 UI 中用显式的垂直虚线画出同步栅栏（barrier line），贯穿同组所有核的泳道，让"对齐"这一抽象概念可视化。

### 4.3 数据量与性能的权衡

仓库代码以 C++（43.8%）和 Python（55.4%）为主。泳道图的前端实现很可能基于 Web 技术（JSON 数据 + JS 渲染）。关键性能考量：

- **64 个同步点 × 每核多个 task**：数据量不大，但渲染同步连线（SVG 弧线）在大规模场景下可能成为瓶颈
- 建议使用 Canvas 而非 SVG 渲染同步连线，或使用 WebGL overlay
- 同步数据的内存开销（Machine 模块为每个 task 分配 64 个同步时间戳）在 UX 层面应提供"采样率"控制：全量采集 vs 采样采集，让用户在精度和开销之间权衡

## 5 竞品参考与最佳实践

### 5.1 Chrome DevTools Performance Panel

Chrome 的性能面板将多线程活动展示在泳道图中，线程间的依赖通过"发起者"（initiator）箭头连接。PyPTO 可参考其做法：

- hover 某个 task 时高亮其 initiator（sync 依赖来源）
- 按住 Shift 选中多个 task 时，显示它们之间的完整依赖图

### 5.2 NVIDIA Nsight Systems

NVIDIA 的 GPU profiling 工具在时间线视图中用箭头展示 kernel 之间的依赖关系，支持按 stream 分组和过滤。PyPTO 可参考其：

- 按 mix group 过滤的侧边面板
- "关键路径"高亮功能（自动标出决定总耗时的最长依赖链）

### 5.3 Intel VTune

VTune 的线程交互视图将同步等待用不同颜色的 bar 内联展示，不需要独立模式。这启发了 3.1 节"第一层"的设计 —— 在不打开 mix 模式时也能看到同步等待占比。

## 6 实施优先级建议

| 优先级 | 改进项 | 价值 | 实施复杂度 |
|-------|-------|------|-----------|
| P0 | Task bar 内的计算/等待颜色区分 | 解决核心痛点 — 看见真实计算耗时 | 低 — 只需 JSON 新增字段 + 前端着色逻辑 |
| P0 | 同一 mix group 的 task 颜色统一 | 解决分组识别问题 | 低 — 读取 mix_groups 配色 |
| P1 | 1C2V 排布 + 分组框 | 缩短视觉扫描距离 | 中 — 泳道排序与布局重构 |
| P1 | 核间同步连线 | 依赖关系可视化 | 中 — 弧线渲染 + 数据映射 |
| P2 | 双击展开核内流水 | 深入分析 | 中 — 子视图渲染 + 导航 |
| P2 | Hover 高亮 + Tooltip | 交互细节 | 低 — 事件监听 + 弹窗 |
| P3 | 关键路径高亮 | 高级分析 | 高 — 需要图算法计算 |
| P3 | 同步线密度控制 / 虚拟化渲染 | 大规模场景性能 | 高 — 渲染优化 |

## 7 总结

Issue #1121 的核心 UX 问题是**信息缺失导致的分析无力感**：用户盯着泳道图却无法区分"真正干活的时间"和"等别人干完的时间"。

结合 PyPTO 的分层架构特点（Tensor/Tile/Block 三级抽象、MPMD 执行模式、IDE 工具链集成），解决方案应遵循以下原则：

1. **渐进式披露**：不打开 mix 模式也能嗅探到同步问题（P0 层），打开后看到完整拓扑（P1 层），深入后看到核内流水（P2 层）
2. **空间映射硬件拓扑**：1C2V 排布让泳道的空间结构与 Ascend 硬件物理拓扑对应，降低心智映射成本
3. **向后兼容**：JSON 格式向后兼容，无 mix 数据时自动降级为经典模式
4. **性能意识**：64 同步点 × 多核多 task 的数据规模需要前端做虚拟化和按需渲染
5. **不只是"看到"，更要"理解"**：最终目标是让用户从泳道图中直接得出"该优化哪个核的哪段计算"的结论，而非仅仅"看到了同步线"
