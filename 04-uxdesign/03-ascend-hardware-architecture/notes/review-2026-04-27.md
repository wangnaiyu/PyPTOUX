## 0. 总体判断

`interaction-spec.md` 整体方向是对的：可信度三态线型、claim toggle、三视图 segmented control、AIC/AIV 经 GM 的硬约束、不写 buffer 容量数字——这些设计决策都准确地反映了上一轮调研的事实边界。

主要问题集中在：

- 个别硬件事实在耦合 / 分离架构上不分场景地复用，导致部分路径在耦合架构下不成立。
- `SDMA` 等本身 verified 的硬件单元被打了 article claim 标签，可信度分级不够细。
- 算子开发者最关心的"软件可见性 / 可分配范围"等场景在详情卡里缺失。
- 缺少型号 → 架构的入口和场景 × 视图的兼容矩阵，用户路径有断点。

下面按 P0 / P1 / P2 优先级列出共 20 条修订建议。

---

## 1. P0 必须改（事实错误，画出来会被打脸）

### A1. 耦合架构 Cube 路径不能含 `FixPipe`

- 当前 §4.2 写：`GM → L1 → L0A/L0B → Cube → L0C → FixPipe → GM/L1`，并标"耦合架构、分离架构均可展示"。
- 但 [overview.md](../../../02-knowledge/00-shared/ascend-aicore-hardware/overview.md) §存储单元把 `FP Buffer / Fixpipe Buffer` 明确列为"分离架构中由官方文档列为新增 Buffer"。
- `FixPipe` 作为搬运/随路处理单元的命名和能力是 V220（分离架构）才提出的；耦合架构 V100/V200 上 `L0C → UB/GM` 走的不是同一通路、不叫 `FixPipe`。
- **修订建议**：把 §4.2 拆成"耦合 Cube 路径"和"分离 Cube 路径"两条，让 `FixPipe` 只出现在分离侧；或在耦合 Cube 路径里把 `FixPipe` 节点替换为通用的 `L0C → UB/GM` 通路并在备注里说明该代际无独立 FixPipe 名义。

### A2. `SDMA` 不能完全归 PTO claim 名下

- 当前 §4.5 把整条 `local GM → SDMA/URMA → remote GM` 标 `article claim`，[overview.md](../../../02-knowledge/00-shared/ascend-aicore-hardware/overview.md) §计算与搬运单元里 `SDMA` 也标了 `article claim`。
- 实际：`SDMA` 引擎本身是独立于 PTO 的硬件 DMA，CANN 官方一直承认其存在，只是"PTO ISA 借 SDMA 做异步后端"才属于 claim。
- 后果：用户会误以为 `SDMA` 是 PTO 的虚构概念。
- **修订建议**：在画布里把 `SDMA` 独立成 verified 实线通路对象（GM ↔ GM 直传，绕过 UB），同时保留场景层的"PTO 借 SDMA 做异步"为 claim。两件事可信度不一样，不要混。

### A5. 910B 集合通信路径要拆成两层可信度

- 当前 §4.6 把 910B "AIC 计算后写 GM、AIV 组合搬运和 Reduce" 与 950 CCU 路径都归 `article claim`。
- 但 §4.3 "AIC/AIV 协作"的 `AIC → GM → AIV` 通路本身已是 `verified`，被 article claim 的是"AIV 自己组合集合通信"这一**实现方式**。
- **修订建议**：§4.6 拆两层——底层通路复用 §4.3 verified 主图；上层"AIV 组装集合通信" / "950 CCU 卸载"标 claim。视觉上保留一条 verified 实线 `AIC → GM → AIV`，叠加灰色虚线"集合通信组装层"。

### B1. 视图切换的"场景 × 视图"适配矩阵缺失

- 用户在分离架构下选了"AIC/AIV 协作"，切到耦合架构后这条场景不存在；spec 当前没说 fallback 行为。
- 类似情况还有：耦合视图下选"集合通信/CCU"会发生什么？
- **修订建议**：补一张 `场景 × 视图` 适配表，给出明确 fallback 策略（自动切到该视图下默认场景，或场景按钮显式置灰 + tooltip 解释为什么不可用）。建议草表：

  | 场景 \ 视图 | 耦合架构 | 分离架构 | 950 演进 |
  | --- | --- | --- | --- |
  | Vector | ✅ | ✅ | ✅ |
  | Cube | ✅ | ✅ | ✅ |
  | AIC/AIV 协作 | 置灰 | ✅ | ✅ |
  | PTO 同步 | ✅ | ✅ | ✅ |
  | PTO 异步 | claim | claim | claim |
  | 集合通信/CCU | 置灰 | claim | claim |

## 2. P1 建议改（影响理解但不致命）

### A3. 中央画布里 `AI Core` 与 `AIC/AIV` 不能并列

- §2 布局文字写"`GM / L2 / AI Core / AIC / AIV / IO-Die`"，三者并列会让人误以为是同级。
- 实际：分离架构下 **AI Core = AIC + AIV 的组合**（[overview.md](../../../02-knowledge/00-shared/ascend-aicore-hardware/overview.md) §耦合模式与分离模式："AI Core 核数以 Cube Core 为准"）。
- **修订建议**：分离架构视图里画一个外框 `AI Core`，内部嵌 `AIC` 和 `AIV` 两个子容器；或在画布对象列表里写成 `AI Core (= AIC + AIV)`。

### A4. AIC : AIV 配比完全没体现

- 调研发现 910B4 是 1 AIC : 2 AIV（arXiv 2505.15112），910B 早期 SKU 可能是 1:1，后续可能 1:N。
- 当前 spec 默认画法暗示 1:1，对算子开发者和优化专家不友好。
- **修订建议**：分离架构子容器里用 "1 AIC + N AIV (示例)" 形态，或在右侧详情卡 AIC/AIV 节点上加"配比"字段，标注"参考型号 910B4：1 AIC : 2 AIV"。

### A8. 缺 `Scalar` 的画布表达

- §2 画布对象列表完全没列 `Scalar`，§3.2 视觉编码也没给 `Scalar` 配色。
- 但 §2.1 视图列表里"耦合架构"写了"共享 Scalar"——对象不在画布上但在视图描述里出现，是缺漏。
- 分离架构下 AIC / AIV 各有独立 Scalar 是关键差异（[overview.md](../../../02-knowledge/00-shared/ascend-aicore-hardware/overview.md) §计算与搬运单元）。
- **修订建议**：补 `Scalar` 进画布对象、视觉编码、详情卡；按"调度单元"风格画（小尺寸、不抢戏）。

### B2. claim toggle 关闭时场景按钮要联动置灰

- §7.1 只说"隐藏 URMA、CCU、PTO 异步虚线路径"。
- 但场景列表里的 PTO 异步、集合通信/CCU 在 claim 关闭后仍然可点 → 点了之后用户看到空图会困惑。
- **修订建议**：claim 关闭时，左侧场景列表里 claim 类场景按钮也置灰，hover 时 tooltip 说明"已隐藏 claim 层，开启后可查看"。

### B5. "画图备注"字段不要渲染到 UI

- §5 详情卡模板里有 `画图备注: 放在 AIV/Vector 侧，不要放入 AIC/Cube 侧` 这种字段。
- 这是给画图者（Claude）看的元数据，最终用户看到这种字段会困惑。
- **修订建议**：`画图备注` 只能存在于源数据 JSON 里，不渲染到 UI；UI 上需要的是"位置说明"或者干脆不显示。

### C1. 详情卡缺"软件可见性 / 可分配范围"

- 算子开发者写 Ascend C kernel 第一个会撞上的事实是：AIV 的代码不能 alloc tensor 在 L1（[overview.md](../../../02-knowledge/00-shared/ascend-aicore-hardware/overview.md) §存储单元 UB 在 AIV 侧、L1 在 AIC 侧）。
- 当前详情卡只说"用途"和"归属"，没说"软件可见性"。
- **修订建议**：详情卡加"软件可见性 / 可分配范围"字段。例：
  - UB → "AIV 代码可分配，AIC 代码不可见"
  - L1 → "AIC 代码可分配，AIV 代码不可见"
  - GM → "AIC / AIV 均可见"

### C2. 顶部缺型号入口

- 用户提到的概念是"310"、"910"这些产品名，不是"耦合架构"、"分离架构"。
- 当前 spec 要求用户先理解架构模式才能选对视图。
- **修订建议**：两选一：
  1. 在顶部加型号选择器，选 `310 / 910 / 910B / 950` 后自动切到对应架构视图。
  2. 在视图切换 tab 上加副标题（如"耦合架构（310 / 910 类）"、"分离架构（910B / A2 / A3 类）"）。
- 优先方案 2，改动小。

### C3. 缺核内累加场景（`L0C` 自反馈）

- 仓库 [block-graph.md](../../../02-knowledge/01-block-level-coding/block-graph.md) 已经在用 `A_MULACC_B` opcode；累加场景里 `L0C` 既是 Cube 输入也是输出，**不出 AICore**，是算子开发最常见的优化模式。
- 当前六个场景一条都没列。
- **修订建议**：在 §4 补"Cube 累加路径"作为 verified 路径，或在 §4.2 详情说明里点出 `L0C` 可作 Cube 输入。

### C4. 缺独立的"GM ↔ GM 直传 (SDMA)"场景

- 与 A2 配合：`SDMA` 既然要独立成 verified 通路，就该有自己的场景入口。
- 性能优化专家想看大块 GM 搬运的延迟/带宽特征时会用到。
- **修订建议**：场景列表加"GM ↔ GM 直传 (SDMA)"，verified 实线，路径就是 `GM → SDMA → GM (本卡或跨卡)`。

## 3. P2 可选优化

### A6. `MTE3 → L1` 标 partially verified

- [overview.md](../../../02-knowledge/00-shared/ascend-aicore-hardware/overview.md) 自己写："`MTE3: UB → GM`；术语表也提到可负责 `UB → GM/L1`，需按具体架构区分。"
- spec §4.1 Vector 路径只画 `UB → GM`，没体现这个歧义。
- **修订建议**：在画布上把 `UB → L1` 画成可选支路并标 `partially verified`。

### A9. 补 `L1 → BT Buffer` 通路

- [overview.md](../../../02-knowledge/00-shared/ascend-aicore-hardware/overview.md) §计算与搬运单元："`MTE1` ... 分离架构还涉及 `L1 → BT Buffer`"。
- spec §4 所有路径都没有这条线。BT 进了画布就该有去路。
- **修订建议**：分离架构 Cube 路径里补 `L1 → BT Buffer (via MTE1)`。

### B3. "路径矩阵"组件定义不清楚

- §2 文字布局里写"底部: 路径矩阵 / legend / 当前路径"，但 §3 §4 §7 都没解释"路径矩阵"长什么样。
- **修订建议**：明确这是"全部场景路径的对照表"还是"当前路径的步骤化展示"。建议是后者，且与 §3.3 路径高亮 + 播放呼应。

### B4. 通路选中详情卡模板缺失

- §5 模板示例都是节点（UB、CCU），没有"选中通路（MTE2、FixPipe）"的模板。
- 而 §3.3 又说"hover 任意通路时高亮起点终点和右侧详情卡"。
- **修订建议**：通路选中详情应该有自己的字段：方向、源、目的、所属架构、对应 PyPTO opcode（如 `COPY_IN` / `L1_TO_L0A`）。

### B6. 增加"对比模式"

- 三张视图通过 segmented control 切换，一次只能看一张。但产品/教学用户最想要的是并排对比耦合 vs 分离。
- **修订建议**：增加一个"对比"模式，左右各一张视图，选中场景在两边同步高亮（场景在某侧不存在就置灰对应侧）。优先级中等。

### B7. hover / select / playing 三态规则

- §3.3 说"hover 任意通路时高亮"，§7.1 又说"播放状态当前路径按顺序高亮"。
- 三种状态如何叠加？hover 优先还是 select 优先？播放时 hover 是否禁用？
- **修订建议**：补 hover / select / playing 三态优先级和叠加规则。

### C5. 右侧加"本图为单核视图"提示

- 性能优化专家会问"我有 24 AIC + 48 AIV，怎么分活？"
- 当前 spec 只画单核内部，没有核间维度。
- **修订建议**：在右侧详情卡或 footer 提一句"本图为单 AICore 内部视图，多核间协作不在此呈现"。

## 4. 已经准确 / 设计合理的部分（建议保留）

业务准确性侧：

- §6 "不把 `FP Buffer` 等同于 `FixPipe`" — 这个区分非常到位。
- §6 "不把 `L2 Cache` 放进单个 AIC 或 AIV" — 对。
- §4.3 AIC/AIV 协作必经 GM、约束"不出现本地 buffer 直连" — 对。
- "不写具体 buffer 容量数字" — 对，符合上一轮 U1 共识。

交互合理性侧：

- 三种可信度线型（实线 / 半实线 / 虚线）+ 节点样式区分清晰，符合 §1.1 "明确区分 verified / partially verified / article claim" 的目标。
- claim toggle 默认开启 + 灰色虚线显示，是稳妥的设计（不直接隐藏，让读者知道"还有未确认层存在"）。
- §6 "必须避免的错误"清单是高质量的护栏。

用户场景覆盖度侧：

- 三类目标用户的画像区分清晰（§1.2）。
- 验收点 §7.2 "10 秒看出耦合 vs 分离"作为关键 KPI 设计得好。
- "关闭 claim 后图仍然完整可信"作为验收点，体现了 verified 主图自洽的工程克制。
