1. 产品判断

Fusion Performance Diagnosis 的第一期不应被定义为 Memory Viewer，也不应被做成大而全 Trace Viewer。 它的核心价值是把一个慢 task 自动解释成可行动的根因假设、证据强度和下一步 PyPTO 修改动作。

核心用户问题 看得到 task 慢，但不知道是计算、等待、依赖、UB pressure、spill-like 行为还是额外 GM/UB 搬运导致。
产品机会 把 PyPTO Toolkit 从执行可视化工具推进为融合算子性能诊断入口。
设计原则 先归因，再下钻；痛点绑定动作；建议必须能被下一轮运行验证。

2. 背景与问题

当前用户通过 debug_options={"runtime_debug_mode": 1} 采集运行上下文，同一轮得到程序结果与 merged_swimlane.json。这套机制可以让用户看到执行时间线，却没有把异常时间、内存压力、等待关系、 PMU counter、program.json、计算图和源码动作串成诊断链路。

设计研究显示，用户情绪谷底并不发生在采集或打开泳道图时，而是发生在“已经看到慢 task，却只能靠经验猜根因”的阶段。 竞品分析进一步说明，成熟性能工具的优势不是图表数量，而是选择窗口、时间线、资源指标、事件详情和行动出口之间的联动。

现状痛点 用户真实意图 产品化转译
打开泳道图后需要手工找慢 task、读依赖、看 copy 节点。 先知道哪几个 task 最值得看，以及主导瓶颈是什么。 提供 Top abnormal tasks / wraps 与 bottleneck type。
task bar 只有总耗时，用户无法区分干活、搬运和等待。 知道时间花在 compute、memory move、wait 还是未解释区间。 提供 compute / memory move / wait / unaccounted 分段泳道。
怀疑 UB pressure 或 spill-like 行为，但没有证据。 确认 peak live memory、UB headroom、额外搬运和贡献对象。 提供 Memory Pressure 下钻与 Top contributors。
试错调 tile、nbuffer 或融合边界后，很难证明因果。 用下一轮运行证明动作是否有效。 提供 Before / After 诊断对比和调优记录沉淀。

3. 目标用户

目标用户：Fusion Performance Engineer

PyPTO / CANN 高阶算子开发与性能调优用户，主要处理大模型 RoPE / Attention 相关融合算子性能交付。 该用户能够阅读泳道图和计算图，也能修改 tile、融合策略或 vec_nbuffer_setting。

他的核心困境不是“不会调”，而是工具没有把性能现象翻译成足够可信、可执行、可验证的根因证据。

非目标用户

  • PyPTO 入门用户：优先需要教程、示例代码和基础错误排查。
  • 纯算法研究者：主要关心算法效果，不负责融合算子性能归因。
  • 应用层调用方：只需要端到端趋势或整体 profiling 报告。
  • 平台管理视角：关注资源利用和调度概览，不修改 tile / nbuffer / fusion boundary。

4. 目标与非目标

产品目标

  • 让用户在打开运行产物后 1 分钟内看到 Top abnormal tasks 与主导瓶颈类型。
  • 把慢 task 的时间拆成 compute、memory move、wait、unaccounted 四类解释。
  • 对内存相关异常给出 peak live memory、UB headroom、extra movement 与 Top contributors。
  • 把诊断结论绑定到 task、wrap、tensor、op、program.json、计算图和源码位置。
  • 让每条建议都能被下一轮运行的 task time、extra movement、UB pressure 或 utilization delta 验证。

非目标

  • 不在第一期做通用 Trace Viewer 或可任意配置 counter 的专家自由探索工具。
  • 不把 PMU CSV 表格直接暴露为产品主能力。
  • 不把 Memory Viewer 作为独立终点，内存信息必须回到诊断和动作。
  • 不承诺自动改写用户 PyPTO 代码，第一阶段只提供证据、建议和跳转出口。
  • 不覆盖所有算子类型，先聚焦融合 task、wrap 与 RoPE / Attention 类高频性能问题。

5. 分期范围

阶段 能力 用户价值 最小数据需求 交付判断
P0 Diagnosis Overview + Top abnormal tasks 打开产物后先建立方向感。 exec_start/end、task duration、wrap mapping、severity rule。 用户能从异常列表进入对应 task / wrap。
P0 Segmented Swimlane 立即区分真计算、额外搬运、等待和未解释时间。 PMU accounted cycles、wait / sync events、memory movement summary。 每个异常 task 至少展示 dominant segment 与 ratio。
P0 Memory Pressure 面板 直接回应 #2141 的 UB pressure、spill-like 与额外 GM/UB 搬运诉求。 live memory、copy in/out、raw tensor memory range、Top contributors。 用户能定位贡献最大的 tensor / op / fusion fragment。
P1 Wait Chain 面板 区分搬得慢、等别人、调度晚和资源未覆盖间隙。 syncEvents、predecessor ready time、schedule wait、peer task mapping。 用户能看到 signal 方、等待方、wrap 和 critical peer。
P1 Pipeline Balance by PMU group 把 PMU group 转译成 C / V / MTE / FixPipe / Scalar 的诊断镜头。 PMU group 2/4/6/7/8 counters、wrap / core mapping、total cycle。 用户能判断 compute-heavy、MTE-heavy、UB-conflict 或 L2 reuse failure。
P2 源码 / 计算图 / program 联动 让诊断结果落到能改的 PyPTO 表达。 rootHash、callOpMagic、leafHash、operand hints、source map。 用户能从诊断跳到源码、计算图节点和 program.json 片段。
P2 Before / After 自动对比 把试错变成验证，沉淀团队可复用调优经验。 run id、task identity matching、metric delta、diagnosis delta。 系统能展示 task time、extra movement、wait ratio、headroom 与 pipeline balance 的变化。

7. 功能需求

FR-01

运行产物自动绑定

系统应识别同一轮运行中的 merged_swimlane.json、program.json、计算图、PMU 文件与源码映射状态，并提示缺失项。

  • 必须展示当前 run id 或可读的运行时间标识。
  • 必须标记数据完整性：complete、partial、missing。
  • 缺失关键证据时，不得给出高置信诊断。
FR-02

Top abnormal tasks / wraps

系统应按 duration、severity、bottleneck type 和 evidence level 展示优先排查对象。

  • bottleneck type 初期至少包括 compute、memory movement、wait、UB pressure、unaccounted。
  • 点击列表项后，泳道图和详情面板必须同步定位到对应 task / wrap。
  • 每条异常必须展示“为什么它异常”的一句话解释。
FR-03

分段泳道图

异常 task bar 应分解为 compute、memory move、wait、unaccounted 四段，并展示各段 absolute time 与 ratio。

  • dominant segment 应可被视觉或排序识别。
  • 点击 segment 后右侧详情只解释当前 segment。
  • unaccounted 不能被隐藏，必须提示证据不足或计量缺口。
FR-04

Memory Pressure 下钻

系统应针对内存相关异常展示 live memory 曲线、UB headroom、extra GM/UB movement、copy edge 与 Top contributors。

  • Top contributors 至少按 tensor、op、fusion fragment 或 task 归因。
  • 每个 contributor 必须能追溯到数据来源字段或关联对象。
  • 建议动作应明确指向 tile、nbuffer、fusion boundary 或中间结果减少。
FR-05

Wait Chain 下钻

系统应区分 predecessor wait、schedule wait、sync wait、resource / uncovered gap，并高亮信号方、等待方和 peer task。

  • 等待关系应支持从当前 task 跳到 critical peer。
  • 当等待证据不足时，应提示无法区分等待类型。
  • 等待诊断应避免与 memory movement 诊断互相覆盖。
FR-06

Pipeline Balance 诊断

系统应把 PMU group 转译为 AIC / AIV / MTE / FixPipe / Scalar 等维度的 utilization、dominant counter 和 critical lane。

  • 不得把 PMU CSV 直接作为主体验。
  • 必须解释 MTE-heavy、AIV idle、UB conflict、L2 reuse failure 等诊断标签的证据。
  • 当 counter 覆盖不足时，应降级为“趋势提示”而非强诊断。
FR-07

源码与产物联动

诊断结论应能回跳到 program.json、计算图节点和源码位置，保留 rootHash、callOpMagic、leafHash 等上游 literal。

  • 跳转失败时，应展示可复制的对象标识。
  • 源码动作建议必须说明它影响哪个诊断指标。
  • 不改写上游字段名、trace key 或文件名。
FR-08

Before / After 诊断对比

系统应支持两轮运行的 task identity matching，并比较 task time、extra movement、wait ratio、UB headroom 和 pipeline balance。

  • 匹配不确定时，应显示匹配置信度和候选对象。
  • 对比结果应区分“改善”“恶化”“无显著变化”“证据不足”。
  • 对比结论可沉淀为团队调优记录或后续 agent 输入。

8. 数据需求与诊断契约

本 PRD 不新增上游事实断言，只把现有研究中的字段和产物诉求整理为产品数据契约。真实字段、路径和 schema 需要在进入实现前回到 PyPTO 本地镜像、官方文档或样例数据中校验。

数据域 用途 关键对象或字段 缺失时的降级
Execution timeline 定位 task / wrap、计算 duration 与异常排序。 merged_swimlane.json、exec_start/end、task identity、wrap mapping。 只展示时间线与人工选择，不给出自动严重度。
Program linkage 把诊断回连到 PyPTO 表达和计算图。 program.json、rootHash、callOpMagic、leafHash、operand hints。 保留对象标识，但不显示源码 / program 跳转。
Memory movement 判断额外 GM/UB 搬运、copy edge 与 Top contributors。 copy in/out、GM/UB bytes、raw tensor memory range、movement summary。 只展示 task 分段中的 memory move ratio，不给 contributor 归因。
Live memory 计算 peak live memory、UB headroom 与 pressure trend。 live tensor set、peak bytes、UB capacity reference、tensor lifetime。 不判断 UB pressure，只提示需要补充 live memory 数据。
Wait / sync 区分 predecessor wait、schedule wait、sync wait 与 peer relation。 syncEvents、ready time、schedule wait、peer task mapping。 等待只作为未解释或弱证据段展示。
PMU counters 解释 pipeline balance、MTE-heavy、UB-conflict 等标签。 PMU group 2/4/6/7/8 counters、core mapping、total cycle。 仅显示趋势，不给高置信 pipeline 诊断。
Run comparison 验证调优动作是否有效。 run id、task identity matching、metric delta、diagnosis delta。 允许人工选择两轮任务后做局部对比。

诊断输出必须显式区分 evidence level。缺少 memory、wait 或 PMU 证据时，系统应降级表达为“可能原因”或“证据不足”， 避免把推断包装成事实。

9. 成功指标与验收标准

指标 衡量方式 期望方向
首次定位时间 从打开运行产物到识别 Top abnormal task 与主导瓶颈的时间。 显著短于人工读泳道图。
诊断覆盖率 异常 task 中可给出 bottleneck type 和 evidence level 的比例。 P0 覆盖 memory movement / UB pressure 相关问题。
建议可执行率 诊断建议能绑定 tile、nbuffer、fusion boundary、tensor/op/source 的比例。 减少“泛泛建议”。
验证闭环率 完成修改后能通过下一轮运行看到 delta 的问题比例。 P2 后成为默认闭环。
证据可信度 用户是否认可诊断证据足以支撑下一步修改。 高于纯 timeline / CSV 观察。

PRD 验收标准

  • P0 完成时，用户可以从异常列表进入分段泳道和 Memory Pressure 面板，完成一次“慢 task -> 内存证据 -> 调参方向”的诊断。
  • P1 完成时，用户可以区分 memory movement、wait chain 与 pipeline imbalance，减少误把等待当搬运或误把搬运当计算瓶颈。
  • P2 完成时，用户可以把诊断跳回源码 / 计算图 / program.json，并通过两轮运行对比验证动作效果。

10. 风险与开放问题

风险：证据链不完整导致误诊

如果 memory、wait、PMU 或 source map 缺失，强行给出单一根因会伤害信任。需要把 evidence level 作为一等输出。

风险：默认视图变成复杂 Trace Viewer

竞品启发应服务于诊断闭环，而不是堆叠 tracks 和 counters。P0 默认视图必须少而强。

开放问题：UB capacity 与 headroom 如何标准化

需要确认不同硬件、task 类型和采集粒度下，UB headroom 的计算口径与阈值。

开放问题：task identity matching 的稳定键

Before / After 对比依赖跨轮匹配，需要确认使用 task id、wrap、hash、op path 还是组合键。

开放问题：建议模板的边界

第一期建议应偏“方向 + 证据 + 验证指标”，还是可以给出更具体的 tile / nbuffer 参数候选，需要结合真实样例验证。

开放问题：source map 缺失时的行动出口

如果源码位置不可用，是否以 program.json、计算图节点、tensor/op 名称作为临时出口，需要产品和实现共同确认。
