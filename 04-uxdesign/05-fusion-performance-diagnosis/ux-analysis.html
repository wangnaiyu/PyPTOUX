Before & After

One-Line Insight / 一句话结论

当前旅程并不缺“产物生成”，缺的是从 merged_swimlane.json 到可操作调优动作之间的诊断桥梁。 目标旅程不是让用户多看几张图，而是把慢 task 自动解释为计算、调度、依赖、内存压力或额外搬运等可行动原因。

核心变化：可视化 -> 诊断闭环

用户业务

大概率是大模型 RoPE / attention 相关融合算子开发与性能交付，处在 PyPTO 高阶调优阶段。

当前机制

runtime_debug_mode 运行前配置，同一轮运行产出程序结果和泳道图文件。

核心痛点

看得到 task 慢，但不知道是不是 UB 压力、额外搬运或哪个 tensor/op 造成。

目标体验

运行后自动归因，定位 task/op/tensor/source，并给出 tile、融合、nbuffer 的下一步建议。

Before & After Journey / 旅程对照

Before 和 After 不强行保持相同步数。能匹配的行为按同一列附近对齐；After 新增的诊断行为被拉宽， 对应 Before 中“观察后只能猜测”的同一问题阶段。

用户行为

打开 Toolkit 后确认系统识别到最新运行产物。

Task

绑定 merged_swimlane.json、program.json、源码和 semantic label。

不再担心打开错轮次或漏看关联产物。

3. 先给性能总览与归因

用户行为

先看最慢 task、瓶颈类型和证据强度。

Task

快速判断异常更像 compute、schedule、dependency、memory movement 还是 UB pressure。

Mindset

先获得方向，再决定是否下钻。

Value把“看图猜”前移为“工具先归因”。

4. 下钻内存压力

用户行为

查看异常 task 的 peak live memory、UB headroom、额外搬运和 Top contributors。

Task

Mindset

从“怀疑 spill”变成“有证据可讨论”。

Value把 UB/spill 从经验判断变成诊断证据。

用户行为

从诊断结论跳到源码、计算图和调参建议。

Mindset

行动方向明确，试错范围缩小。

Value诊断直接落到用户能改的 PyPTO 表达。

用户行为

修改后再次运行，并查看新旧结果对比。

Task

Mindset

从试错转为验证，每轮修改都有因果记录。

Value证明性能提升是否来自 UB 压力下降。

Change Summary / 增删改摘要

After 不是简单增加 UI 控件，而是把原本堆在用户脑内的判断拆成工具可见、可验证、可复用的阶段。

保留

保留 runtime_debug_mode 运行采集入口，保留 merged_swimlane.json 作为核心可视化产物。

改造

把“手工找文件、手工绑定上下文、手工读图归因”改成“Toolkit 自动识别本轮运行与关联产物”。

新增

新增性能总览归因、内存压力诊断、Top contributors、源码回跳、调参建议和再次运行对比。

弱化

弱化专家肉眼猜测与盲目试错，让 UB spill / 额外搬运从经验假设变成结构化证据。

设计原则

这个场景的核心用户是高阶算子开发/调优用户。After 旅程要围绕他们最真实的目标设计：更快定位根因，更稳选择下一步动作。

不要让用户成为泳道图侦探。让工具把执行现象翻译成 PyPTO 开发动作。

Target experience statement
先归因，再下钻

用户进入性能分析时，先看到瓶颈类型和证据强度，再决定看 task、tensor、copy 或源码。

痛点必须绑定动作

每一个痛点都落到具体阶段：打开产物、观察异常、猜测根因、试错验证，而不是作为散落的用户抱怨。

建议必须可验证

每条建议都应该能在下一轮运行中被对比验证，例如 task time、额外搬运、UB pressure 是否改善。

设计价值

这个 After 旅程的价值不在“阶段更多”，而在每一步都有证据、方向和验证方式，让专家用户更安全地推进性能交付。

降低专家认知负担

  • 把最消耗心力的“猜测 UB / 搬运根因”变成工具先给证据。
  • 让用户从读图侦查转向判断和决策，情绪谷底被抬高。

强化 Toolkit 核心定位

  • 从执行可视化升级为性能诊断入口。
  • 把 issue #2141 从单点指标需求扩展成 Fusion Performance Advisor 能力雏形。

支撑 IDE 与 agent 自动化

  • 结构化输出瓶颈类型、内存压力、Top contributors 和调参建议。
  • 为性能 skill、自动调优和跨轮对比提供稳定数据接口。

沉淀可复用调优知识

  • 有效动作可以沉淀成团队规则，而不是停留在个人经验和截图里。
  • 同类 RoPE / attention 融合片段再次出现时，可以更快复用诊断链路。

该用户已经跨过基础使用阶段，能够阅读泳道图和计算图，也能修改 tile、融合策略或 vec_nbuffer_setting。 他的核心困境不是“不会调”，而是现有工具没有把性能现象翻译成可验证的根因证据。

Needs

非目标用户

这些人群不是当前诊断闭环的优先服务对象，避免把高阶性能诊断能力稀释成泛化教程或通用 profiling 报告。

PyPTO 入门用户

刚开始学习 PyPTO，主要需要概念解释、示例代码和基础错误排查。

纯算法研究者

主要关心模型算法正确性和效果，不直接负责融合算子性能归因。

应用层调用方

只需要整体 profiling 报告或端到端性能趋势，不需要下钻到 tensor/op/fusion fragment。

平台管理视角

更关注资源利用率和任务调度概览，而不是修改 tile、nbuffer 或融合边界。
