# Fusion Performance Diagnosis Demo Story

## 场景定位

这是一条面向 issue #2141 的 demo story，用来支撑后续 Fusion Performance Diagnosis 原型或评审演示。故事关注的是高阶 PyPTO 融合算子调优用户如何从“看见慢 task”走到“确认 UB pressure / 额外 GM-UB 搬运 / AIC-AIV 等待链路，并形成下一步调参动作”。

本故事使用 `A5 / Ascend 950` 单 die 口径设计泳道数量：18 个 core cluster，每个 cluster 包含 1 条 AIC 泳道和 2 条 AIV 泳道，共 54 条硬件执行泳道。该口径来自 `pto-isa/docs/reference/pto-cvid-cluster-id-mapping.md` 中的 A5 CVID mapping 说明。demo 数据为 L2 schema-generated / share-safe，不伪装成真实运行结果。

## 用户与任务

用户是 Fusion Performance Engineer，正在优化一个 RoPE / Attention 融合片段：

- 业务目标：降低 decode 阶段 `interleave_rope + attention` 融合链路的单 step latency。
- 当前现象：打开 `merged_swimlane.json` 后，用户能看到 `wrapId = 7` 附近的 AIV task 明显变长，但不能确定是 Vector 计算慢、UB live set 过高、额外 GM/UB 搬运，还是在等待 peer AIC。
- 用户期望：工具直接给出异常 task、证据强度、内存压力来源、等待链路和可验证的下一步动作。

## Demo 故事

### 1. 捕获：用户保留熟悉的采集方式

用户在 PyPTO 程序中启用 `debug_options={"runtime_debug_mode": 1}`，运行 `interleave_rope_attention_fuse`。这一步不改变现有工作流，仍然产出可被 Toolkit 消费的泳道图、PMU 和图关联材料。

在 demo 中，这一步对应数据包里的：

- `scenario.issue`: `#2141`
- `scenario.operator`: `interleave_rope_attention_fuse`
- `sourceBasis`: 记录 schema 来源、硬件泳道口径和生成规则

### 2. 总览：工具先指出最值得看的 task

用户打开 Toolkit 后，不再从空白泳道图开始找线索，而是先看到 Diagnosis Overview：

- Top abnormal task: `A5D0C07V0-T023`
- 所在泳道：`die0.cluster07.AIV0`
- 主导瓶颈：`memory_movement`
- 伴随瓶颈：`wait_chain`
- evidence level：`L2-schema-generated / high within demo`

系统解释：

> `wrapId = 7` 的 `interleave_rope_split_kv` task 总耗时 142.8 us，其中 memory move 56.1 us，wait 36.5 us。该 task 的 UB 诊断阈值余量只剩 13.5 KiB，并出现额外 GM -> UB reload。

### 3. 泳道：54 条硬件执行泳道真实展开

用户切到泳道图时，看到 A5 单 die 的 54 条执行泳道全部存在，而不是只展示有异常的几条：

- `die0.cluster00.AIC` 到 `die0.cluster17.AIC`
- `die0.cluster00.AIV0` / `AIV1` 到 `die0.cluster17.AIV0` / `AIV1`

默认视图可以按 cluster 折叠，但数据层必须保留完整 `lanes`。异常区间在 `cluster07.AIV0` 高亮，peer `cluster07.AIC` 和 `cluster07.AIV1` 被联动标记。

### 4. 分段：慢 task 被拆成 compute / memory move / wait / unaccounted

用户点击 `interleave_rope_split_kv` 后，右侧看到分段解释：

| segment | duration_us | ratio | 解释 |
| --- | ---: | ---: | --- |
| compute | 38.4 | 26.9% | Vector 主计算仍在合理范围 |
| memory_move | 56.1 | 39.3% | GM -> UB reload 与 UB -> GM spill-like copy 占比最高 |
| wait | 36.5 | 25.6% | AIV0 等待 peer AIC 完成 QK matmul 后的同步点 |
| unaccounted | 11.8 | 8.3% | 证据不足，保留为未解释 |

这个瞬间的产品价值是把“我猜是不是 UB 压力”变成“有证据的异常分段”。

### 5. 内存压力：定位 top contributors

Memory Pressure 面板显示：

- `ubBudgetBytes`: 262144，标注为 demo 诊断阈值，不声明为硬件 UB 容量事实。
- `peakLiveBytes`: 248320。
- `ubHeadroomBytes`: 13824。
- Top contributors:
  - `rope_cache_split.k_cache_tile`
  - `rope_cache_split.v_cache_tile`
  - `interleave_rope.tmp_sin_cos`
  - `attention_score.partial_acc`

系统解释：`k_cache_tile` 和 `v_cache_tile` 在 `interleave_rope_split_kv` 的生命周期重叠过长，导致 AIV0 在同一 task 内出现额外 GM -> UB reload。

### 6. 等待链路：确认谁让谁等

Wait Chain 面板显示 `CV_SYNC_WAIT` / `CV_SYNC_SET` 关系：

- waiting task: `A5D0C07V0-T023`
- signal task: `A5D0C07C-T019`
- peer lane: `die0.cluster07.AIC`
- sync type: `CV_SYNC_WAIT`
- wait duration: 36.5 us

系统说明等待不是数据搬运本身；AIC/AIV 数据仍按 `GM/L2` 或 trace 证据解释，`CV_SYNC_WAIT` 只表达控制依赖。

### 7. 行动：把诊断落到 PyPTO 修改动作

系统建议用户优先尝试两类动作：

1. 调小 `interleave_rope` 的 K/V tile，使 `k_cache_tile` 和 `v_cache_tile` 的 live range 错开。
2. 检查 `vec_nbuffer_setting`，避免 AIV 侧过度合并导致 UB headroom 过低。

每条建议都绑定验证指标：

- `task_time_us` 是否下降
- `extra_gm_ub_bytes` 是否减少
- `ubHeadroomBytes` 是否增加
- `wait_ratio` 是否下降

### 8. 验证：再次运行后形成可复用结论

用户按建议调小 K/V tile 并降低 AIV 侧合并强度后再次运行。demo 的 `beforeAfter` 区块给出验证结果：

- target task 从 142.8 us 降至 101.6 us
- extra GM/UB movement 从 384 KiB 降至 160 KiB
- UB headroom 从 13.5 KiB 增至 58 KiB
- wait ratio 从 25.6% 降至 14.2%

最终故事闭环是：用户不再把调优结论停留在截图和经验里，而是获得一条可追溯、可复跑、可沉淀的诊断记录。
